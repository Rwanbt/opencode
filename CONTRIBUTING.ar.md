# المساهمة في OpenCode

نريد تسهيل مساهمتك في OpenCode. إليك أكثر أنواع التغييرات التي يتم دمجها شيوعاً:

- إصلاحات الأخطاء
- LSPs / منسّقات إضافية
- تحسينات أداء LLM
- دعم مزودين جدد
- إصلاحات لخصائص بيئات معينة
- سلوك قياسي مفقود
- تحسينات التوثيق

ومع ذلك، يجب أن تمر أي ميزة واجهة مستخدم أو ميزة منتج أساسية بمراجعة تصميم مع الفريق الأساسي قبل التنفيذ.

## توقعات Pull Request

### سياسة Issue أولاً

**يجب أن تشير جميع PRs إلى issue موجودة.** قبل فتح PR، افتح issue يصف الخطأ أو الميزة. يساعد ذلك المشرفين على الفرز ويمنع العمل المكرر. قد يتم إغلاق PRs بدون issue مرتبطة دون مراجعة.

### لا لجدران النصوص التي يولدها الذكاء الاصطناعي

أوصاف PR و issues الطويلة المولدة بواسطة AI غير مقبولة وقد يتم تجاهلها. احترم وقت المشرفين:

- اكتب أوصافاً قصيرة ومركزة
- اشرح ما تغير ولماذا بكلماتك الخاصة
- إذا لم تتمكن من شرحه بإيجاز، فقد يكون PR الخاص بك كبيراً جداً

### عناوين PR

يجب أن تتبع عناوين PR معايير conventional commit: `feat:` ميزة جديدة، `fix:` إصلاح خطأ، `docs:` توثيق، `chore:` صيانة، `refactor:` إعادة هيكلة، `test:` اختبارات.
